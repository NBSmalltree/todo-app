# TodoFloat — 桌面智能待办清单

## 申报类型说明

**架构类创新 + 方法流程类创新 — 基于 Electron 多窗口架构的悬浮待办管理系统，融合大语言模型实现任务自动分类与工作智能分析，提升了个人任务管理的效率和数据洞察能力。**

## 一、背景介绍

在日常研发和项目管理工作中，任务管理是保障工作效率的核心环节。目前常见的任务管理工具存在以下几类问题：

1. **工具与工作流程脱节**：多数待办清单应用以独立窗口运行，用户需要频繁在任务工具和工作窗口之间切换，打断工作思路。尤其在多屏幕、多任务并行的开发场景中，来回切换窗口的成本不可忽视。

2. **任务数据缺乏深度利用**：传统待办工具仅提供"添加—完成—删除"的基本生命周期，完成的任务往往直接丢弃或仅简单归档，无法从中提炼出工作模式、时间分配、效率趋势等有价值的信息。用户难以回答"这周我主要在做什么""哪类工作投入过多或不足"等关键问题。

3. **任务分类依赖人工**：归档后的任务需要用户手动打标签或分类，增加了操作成本，导致大多数用户放弃分类，历史数据逐渐变成"死数据"，无法被有效检索和分析。

4. **工具碎片化**：任务记录在一个应用，工作日志在另一个应用，时间统计又在第三个应用。数据分散在不同工具中，难以形成统一的工作视图。

针对上述问题，我们设计并实现了 TodoFloat——一款悬浮于所有窗口之上的桌面智能待办清单，通过"始终可见 + AI 自动分类 + 数据驱动分析"的方式，将任务管理融入日常工作流，同时让历史任务数据产生实际价值。

## 二、解决方案

### 1. 概括描述如何实现

TodoFloat 采用 Electron 桌面应用框架，构建了一个**多窗口协作**的架构体系：主窗口以"始终置顶 + 透明悬浮"的方式覆盖在其他应用之上，用户无需切换窗口即可快速记录和管理任务；归档与分析功能通过独立窗口承载，由系统托盘统一入口管理。在此基础上，通过集成 OpenAI 兼容协议的大语言模型 API，实现了任务归档时的**自动分类**和**工作数据智能分析**两个核心能力，将被动的任务记录转化为主动的工作洞察。

### 2. 方案具体实现步骤

#### 整体架构

TodoFloat 采用经典的 Electron 主进程/渲染进程分离架构，整体分为四层：

```
┌─────────────────────────────────────────────────────────┐
│                    渲染进程（React 18）                    │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐     │
│  │  TodoWindow   │ │  TrayView    │ │  Settings     │     │
│  │  悬浮待办窗口  │ │  归档+分析页  │ │  设置页面     │     │
│  └──────┬───────┘ └──────┬───────┘ └──────┬───────┘     │
│         └────────────────┼────────────────┘              │
│                    preload.js (IPC 桥接)                  │
├─────────────────────────────────────────────────────────┤
│                    主进程（Electron）                      │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐     │
│  │   main.js     │ │  database.js │ │   llm.js      │     │
│  │ 窗口管理/IPC  │ │ SQLite 存储  │ │ LLM 集成      │     │
│  └──────────────┘ └──────────────┘ └──────────────┘     │
├─────────────────────────────────────────────────────────┤
│  构建层：Vite 6 + electron-builder + GitHub Actions      │
└─────────────────────────────────────────────────────────┘
```

**前端层**：React 18 + Tailwind CSS 构建三个独立页面组件（TodoWindow、TrayView、Settings），通过 Hash 路由切换，共享同一 preload 脚本暴露的 IPC 接口。

**桥接层**：preload.js 通过 `contextBridge.exposeInMainWorld` 将 Electron 的 IPC 通信封装为 `window.electronAPI` 对象，实现渲染进程与主进程之间的安全通信。

**主进程层**：main.js 负责多窗口生命周期管理和 IPC 事件分发；database.js 封装 SQLite 数据库操作；llm.js 封装大语言模型调用逻辑。

**构建层**：Vite 负责前端资源构建，electron-builder 负责打包安装包，GitHub Actions 实现 macOS/Windows 双平台自动构建与发布。

【图片：应用整体模块交互示意图——展示三个窗口（悬浮窗、归档窗、设置窗）通过系统托盘入口的关系，以及与 SQLite 数据库和 LLM 服务之间的数据流向】

#### 具体实现步骤

**步骤一：悬浮窗口的创建与管理**

应用启动后，主进程执行以下操作：

1. 初始化 SQLite 数据库，创建 todos 表和 settings 表，开启 WAL 模式提升并发读写性能；
2. 从数据库读取上次保存的窗口位置和尺寸（`window_bounds`），校验该位置是否在当前可见屏幕区域内（支持多显示器断开后的自动重置）；
3. 创建 BrowserWindow，配置 `alwaysOnTop: true`、`frame: false`、`transparent: true`、`skipTaskbar: true`，使窗口悬浮于所有应用之上且不在任务栏显示；
4. 设置窗口层级为 `'floating'`（而非更高优先级的 `'screen-saver'`），确保中文输入法候选窗口能正常显示在悬浮窗之上；
5. macOS 下启用 `setVisibleOnAllWorkspaces(true, { visibleOnFullScreen: true })`，使窗口在所有桌面空间和全屏应用中可见；
6. 禁用 Chromium 原生的 Ctrl+滚轮缩放，由渲染进程接管缩放逻辑；
7. 创建系统托盘，注册右键菜单（待办清单 / 历史归档 / 设置 / 退出），点击托盘图标弹出菜单；
8. 应用启动时通过 `app.requestSingleInstanceLock()` 获取单实例锁，若获取失败（已有实例在运行）则立即退出，避免 Windows 下多次双击导致多个进程和多个托盘图标共存的问题。当用户尝试启动第二个实例时，已有实例通过 `second-instance` 事件自动将悬浮窗激活并聚焦到前台。

【图片：系统托盘右键菜单截图——展示"待办清单、历史归档、设置、退出"四个菜单项】

**步骤二：窗口缩放机制**

用户可通过两种方式缩放悬浮窗口：

- **Ctrl/Cmd + 滚轮**：渲染进程监听 wheel 事件，根据 deltaY 值计算缩放增量（鼠标滚轮通常 ±100/120，触控板为较小值），通过 IPC 调用主进程的 `applyScaleToWindow` 方法，保持窗口中心点不变地进行等比缩放。
- **四角拖拽**：在窗口四角放置透明的拖拽热区（16x16 像素），监听 mousedown/mousemove/mouseup 事件，根据对角线方向的移动距离计算缩放比例。

缩放范围限制在 0.3x 至 2.5x 之间，使用 CSS `zoom` 属性实现前端内容的同步缩放，配合 `transition: zoom 0.15s ease-out` 实现平滑过渡。

【图片：窗口缩放效果对比图——同一悬浮窗口在 0.5x、1.0x、2.0x 三个缩放比例下的并排截图，展示等比缩放效果】

**步骤三：任务的 CRUD 与动画系统**

任务的增删改查流程如下：

1. **添加任务**：用户在输入框输入任务内容，按回车提交（通过 `isComposingRef` 检测中文输入法组合状态，避免输入过程中误提交）。主进程执行 `INSERT INTO todos`，返回新记录，前端为其添加 `todo-enter` CSS 动画（opacity 0→1，translateY -8px→0，scale 0.96→1）。
2. **标记完成**：点击左侧圆圈，触发 `todo-check-animate` 脉冲动画（scale 1→1.25→1），主进程更新 `completed` 和 `completed_at` 字段。已完成任务自动排列到底部，中间以分隔线区分。
3. **删除任务**：先播放 `todo-exit` 滑出动画（translateX 0→30px，opacity 1→0，max-height 收缩至 0），等待 200ms 动画完成后才执行数据库删除操作，确保视觉连贯性。
4. **右键菜单与颜色标记**：右键点击任务弹出自定义上下文菜单，未完成任务可选择 4 种紧急程度颜色（🔴紧急、🟠重要、🟡一般、🟢低优），选中颜色后任务左侧显示对应颜色竖条作为视觉标记，再次点击可切换或清除。菜单还提供"标记为已完成/恢复为待办""归档""删除"等操作选项。仅已完成任务允许归档。

5. **拖拽排序**：未完成的任务支持上下拖拽重新排列。每项左侧显示 6 点拖拽手柄（hover 时可见），拖拽时被拖项变为半透明，目标位置显示天蓝色指示线。放下后通过事务批量更新数据库中的 `sort_order` 字段，实现排序的持久化存储。新添加的任务自动排在列表最前。

【图片：悬浮窗口右键菜单截图——对一条已完成任务右键弹出的上下文菜单，展示"恢复为待办""归档""删除"三个选项】

**步骤四：归档与 AI 自动分类**

归档流程是任务数据增值的核心环节：

1. 用户对已完成任务右键选择"归档"，主进程更新 `archived` 和 `archived_at` 字段；
2. 归档完成后，主进程异步检查该任务是否已有分类标签；
3. 若无分类标签且用户已配置 API Key，主进程在后台调用 LLM 分类服务：
   - 使用 `temperature: 0.1` 的低随机性参数，确保分类结果的确定性和一致性；
   - System Prompt 定义了固定的工作类别体系（开发、设计、测试、文档、会议、沟通、运维、学习、管理、其他）；
   - LLM 仅返回一个类别名称，主进程将结果写入 `category` 字段；
4. 分类完成后，通过 IPC 通知归档窗口刷新数据，用户即可看到自动标注的分类标签；
5. 用户也可在归档页面对任意记录手动触发"AI 分类"或手动修改分类。

【图片：历史归档页面截图——展示归档任务列表，包含任务内容、AI 自动标注的类别标签（如"开发""会议""文档"等）、备注、归档时间，以及顶部的搜索/筛选栏和导出按钮】

**步骤五：工作数据分析与可视化**

工作分析模块提供三个时间维度（周/月/年）的数据洞察：

1. **数据聚合**：主进程的 `getWorkAnalysis` 方法根据选定的时间范围，从 SQLite 查询归档任务，按类别和日期进行聚合统计，同时计算全局的完成率（已归档数 / 总任务数）；
2. **前端可视化**：渲染进程使用 Recharts 库绘制类别分布饼图（环形图）和每日任务柱状图，以及类别占比进度条；
3. **AI 智能分析**：数据加载完成后，异步调用 LLM 分析接口。LLM 接收完整的统计数据（任务数量、分类分布、每日分布、完成率等），按照 Prompt 要求生成三个维度的分析——工作重点与时间分配、工作节奏规律、1-3 条优化建议，总字数控制在 500 字以内。结果以 Markdown 格式返回，前端通过 ReactMarkdown 渲染。

【图片：工作分析页面完整截图——展示顶部的周/月/年切换按钮、四张概览卡片（待办任务数、归档任务数、工作类别数、完成率）、左侧类别分布饼图、右侧每日任务柱状图、底部 AI 工作分析建议区域】

**步骤六：多窗口同步与数据一致性**

应用中存在三个窗口（悬浮窗、归档窗、设置窗），数据变更需要跨窗口同步：

1. 归档操作后，主进程向归档窗口发送 `data-changed` 事件通知刷新；
2. 恢复待办操作后，主进程向悬浮窗发送 `data-changed` 事件通知刷新；
3. 主题变更时，主进程遍历所有存活窗口，向每个窗口广播 `theme-changed` 事件；
4. 透明度变更时，主进程向悬浮窗发送 `opacity-changed` 事件。

【图片：多窗口 IPC 事件交互流程图——展示悬浮窗、归档窗、设置窗三个窗口与主进程之间的 IPC 事件流向，标注 data-changed、theme-changed、opacity-changed 等事件的触发场景和方向】

**步骤七：主题系统**

通过 CSS 变量 + `data-theme` 属性实现三套主题的实时切换：

- **浅色模式**（默认）：白色背景，灰色文字；
- **深色模式**：深蓝灰色背景（Catppuccin Mocha 色板），适配暗色环境；
- **护眼模式**：米黄色暖色调背景，降低蓝光刺激。

主题通过 `document.documentElement.setAttribute('data-theme', theme)` 切换，配合 CSS 选择器 `[data-theme="dark"]` 和 `[data-theme="eye-care"]` 覆盖所有 Tailwind 默认样式类。

【图片：三种主题模式对比截图——并排展示同一悬浮窗口在浅色模式、深色模式、护眼模式下的外观效果】

**步骤八：CI/CD 自动化发布**

通过 GitHub Actions 实现一键发布：

1. 开发者执行 `npm run release`，脚本自动更新 package.json 版本号、提交、推送并创建 Git tag；
2. tag 推送触发 GitHub Actions 工作流，在 macOS 和 Windows 环境上并行构建安装包（.dmg 和 .exe）；
3. 构建完成后自动创建 GitHub Release 并上传安装包。

## 三、创新亮点

1. **悬浮窗口 + 透明无框设计**：采用 `alwaysOnTop + transparent + frame: false` 组合，实现了任务清单悬浮于所有应用之上的轻量级体验。用户无需切换窗口即可随手记录任务，将待办管理的摩擦成本降至最低。同时通过 `floating` 层级而非更高优先级，兼顾了中文输入法等系统级浮层的兼容性。

2. **Ctrl+滚轮等比缩放 + 四角拖拽缩放**：绕过 Chromium 原生缩放机制，自行实现了基于 CSS zoom 的窗口内容等比缩放方案。支持鼠标滚轮和触控板两种输入设备的归一化处理，并提供四角拖拽的直觉交互，缩放过程中保持窗口中心点不变，实现了 0.3x 到 2.5x 的自由缩放。

3. **LLM 驱动的任务自动分类**：在任务归档时异步调用大语言模型，基于任务文本自动判断工作类别。采用低 temperature（0.1）参数确保分类确定性，预定义了 10 个标准工作类别。该方案免去了用户手动打标签的成本，让历史数据自动结构化，为后续的工作分析奠定数据基础。

4. **数据驱动的 AI 工作分析**：将归档的任务数据聚合为多维度统计信息（类别分布、每日分布、完成率等），作为上下文传递给大语言模型，生成个性化的工作效率分析报告。报告包含工作重点分析、节奏规律识别和具体优化建议三个维度，实现了从"记录任务"到"理解工作模式"的跨越。

5. **OpenAI 协议兼容的模型抽象层**：AI 模块通过统一的 OpenAI SDK 接口实现，仅需配置 API Key、Base URL 和模型名称三个参数，即可接入 OpenAI、DeepSeek、智谱、通义千问、Claude 等所有兼容 OpenAI 协议的大模型服务，实现了 AI 能力的模型无关性。

6. **窗口位置记忆与多显示器容错**：退出时自动将窗口的坐标和尺寸持久化到 SQLite，下次启动时恢复。同时检测保存的坐标是否在当前可见屏幕区域内，若曾连接的外接显示器已断开导致窗口不可见，自动重置为默认位置，解决了多显示器场景下的窗口丢失问题。

7. **多窗口数据同步机制**：通过 Electron 的 IPC 事件广播实现了悬浮窗、归档窗、设置窗之间的实时数据一致性。归档/恢复/分类等操作完成后，相关窗口自动刷新，避免用户手动刷新或看到过时数据。

8. **全链路自动化发布流程**：一条 `npm run release` 命令完成版本号更新、代码提交、标签推送，触发 GitHub Actions 在 macOS 和 Windows 上并行构建安装包并自动发布到 GitHub Releases。整个发布过程无需手动操作任何平台。

9. **拖拽排序与优先级可视化**：基于 HTML5 原生拖放 API 实现任务的手动排序，无需引入第三方拖拽库。配合数据库 `sort_order` 字段的事务批量更新，实现了轻量且可靠的排序持久化。结合颜色标记功能，用户可以直观地通过拖拽调整任务顺序、通过颜色区分紧急程度，形成"位置 = 优先级 + 颜色 = 紧急度"的双重任务管理维度。

10. **单实例锁机制**：通过 Electron 的 `requestSingleInstanceLock` API 实现应用的单实例运行保障。当用户误操作多次启动时，第二个实例自动退出并将已有窗口激活到前台，避免了多进程竞争数据库、托盘图标重复等 Windows 平台常见问题，提升了应用的健壮性和用户体验。
